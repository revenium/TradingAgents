## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TradingAgents process -> Revenium (api.revenium.ai) | Trace/transaction metadata crosses to an external SaaS over the existing Phase 1 metering channel |
| Environment (REVENIUM_TRACE_TYPE) -> config -> payload | Process-controlled env var becomes an outbound label |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | trace_name / transaction_name / trace_type sent to Revenium | mitigate | Only symbolic values are sent: transaction_name = agent node name; trace_type = config label; trace_name = f"{ticker}-{trade_date}". No secrets, API keys, raw prompts, or PII. trace_name truncated to 200 chars; ticker is already path-sanitized upstream (safe_ticker_component). |
| T-02-02 | Tampering | REVENIUM_TRACE_TYPE env var | accept | Treated as an opaque outbound label only; never interpolated into a path, query, shell, or eval. Process-controlled, not end-user input. Low risk. |
| T-02-03 | Denial of Service | tracing enrichment in on_llm_end / on_chat_model_start | mitigate | All new code stays inside the existing fail-open try/except (# noqa: BLE001); a malformed field or contextvar failure drops the event and never blocks or crashes the trading run. Metering stays fire-and-forget on a daemon thread. |
| T-02-04 | Tampering | trace_id / parent_transaction_id chain to Revenium | accept | UUIDs sent to the same trusted Revenium account already used in Phase 1. No new exposure surface. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this phase (revenium-metering 6.8.1 already VERIFIED:PyPI in Phase 1). No install task, no new supply-chain surface. |
