<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM callback → Revenium API | Token usage + attribution + (potentially) sensitive run metadata cross here via the metering client |
| subscriber PII | subscriber.id is a real email (john.demic+trading@revenium.io) carried on every event |
| Revenium failure → trading run | A metering failure must not be able to halt or corrupt the trading run (fail-open) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | callback.py / client.py fail-open log paths | mitigate | Fail-open warnings log only symbolic names (agent/operation), never the API key or request bodies/prompts (D-06; REVENIUM_CAPTURE_PROMPTS stays off — out of scope); verified by grep for noqa BLE001 wrapping |
| T-02-02 | Information Disclosure | subscriber PII (email) in payload | accept | The email is a deliberate plus-addressed alias controlled by the project owner (D-02), sent only to the trusted Revenium account over TLS; no broader exposure surface in this phase |
